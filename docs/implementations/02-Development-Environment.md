# 02 — Development Environment

BatchPay uses a monorepo because the contract and frontend form one product and evolve together. A single commit can therefore change a contract interface, its tests, the frontend integration, and its documentation atomically.

```text
BatchPay/
├── backend/       Foundry contracts, tests, scripts, dependencies
├── frontend/      reserved for the Next.js application
├── docs/          product and design references
└── docs/implementations/  this handbook
```

Develop from WSL/Linux rather than mixing Windows and Linux commands. The project was initialized under `/mnt/e/WOEKSPACE/BLOCKCHAIN/BatchPay`; commands in this handbook assume a POSIX shell and should be run from the directory stated in each chapter.

## Tooling responsibilities

- Git records design and code changes together and is initialized only at the repository root.
- Foundry compiles, tests, formats, scripts, and interacts with EVM contracts in `backend/`.
- Solidity is the contract language; the current source declares `pragma solidity ^0.8.24`.
- Node.js/Next.js will support the frontend when that scope begins; it is not yet scaffolded in this repository.

Keep secrets outside Git. Root `.gitignore` excludes environment files and frontend build artifacts; Foundry's nested ignore file excludes `backend/out/` and `backend/cache/`. Use `.env.example` for variable names only. A private key, RPC URL containing credentials, and API key must never appear in a commit, terminal paste, or screenshot.

Before coding, run `git status`, then work from `backend/` for Foundry commands. That small habit prevents running `forge` in the monorepo root, where it cannot find `foundry.toml`.

## Reproducibility as a development requirement

Smart-contract source is only useful if another developer can compile the same bytecode and reproduce the test result. The environment therefore has four layers: a known Foundry installation, a compiler version selected by the project or pragma, pinned Git submodules, and source-controlled configuration. Avoid depending on a globally installed OpenZeppelin package, a local RPC endpoint, or editor-specific settings that a reviewer cannot reproduce.

The current source permits compiler versions compatible with `^0.8.24`. Foundry selected Solidity 0.8.35 in the recorded test run. Before a public deployment, pin the exact compiler and optimizer configuration in `foundry.toml`, because those choices affect generated bytecode and explorer verification.

Use an editor with Solidity syntax support, formatter integration, and a visible terminal. Editor extensions improve feedback but must not silently rewrite generated files or insert private configuration into Git. `forge fmt` is the shared formatting authority; run it before commits rather than relying on each contributor's editor.

## Daily workflow

A safe small change has a repeatable loop:

```bash
cd backend
forge fmt
forge build
forge test
cd ..
git diff --check
git status
```

Formatting first prevents review noise. Building catches import and type errors quickly. Tests establish behavior. `git diff --check` detects trailing whitespace and patch mistakes. Finally, inspecting status prevents accidental inclusion of generated artifacts or secrets.

For investigations, start Anvil in a dedicated terminal and keep its output visible. It is a local chain, not a simulation inside the test runner: it is useful for manual wallet and script testing but its state disappears on restart unless explicitly configured. Never point experimental scripts at a funded production account simply because a local endpoint was unavailable.

## Environment boundaries

The frontend's future toolchain should remain independent of Foundry. Node dependencies belong under `frontend/`; contract dependencies belong under `backend/lib/`. Shared facts—contract ABI, deployed addresses, network configuration—should be generated or maintained deliberately, not imported from a build cache. This avoids a common monorepo failure mode where one side works only because it reads unstaged local output from the other.
